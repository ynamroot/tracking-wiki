# Avo (avo.app) 경쟁 분석 리서치 (원자료)

## 1. 푸는 문제 (Problem)

Avo는 행동 데이터 생애주기(설계 → 구현 → 검증 → 운영) 중 **설계(design)와 구현·검증(implementation/validation)을 잇는 구간**을 집중 공략한다. 핵심 문제의식은 "설계된 트래킹 플랜(무엇을 어떻게 수집할지 정한 명세)"과 "실제 코드가 내보내는 이벤트"가 시간이 지나면 어긋난다는 것이다. Avo는 이를 "트래킹 플랜을 단일 진실 원천(single source of truth)으로 삼고, 그 명세를 코드 생성과 런타임 검증으로 강제한다"는 방식으로 해결한다.

- 설계 협업 도구(Tracking Plan)로 명세를 만들고([The Avo Tracking Plan](https://www.avo.app/docs/data-design/avo-tracking-plan))
- Codegen으로 구현 단계의 오류를 줄이고([Codegen overview](https://www.avo.app/docs/implementation/avo-codegen-overview))
- Inspector로 실제 배포 데이터가 명세를 지키는지 검증·운영 감시한다([Inspector overview](https://www.avo.app/docs/reference/avo-inspector-sdks/overview))

즉 **"데이터 품질 거버넌스(governance)"**가 핵심 타깃이며, 특히 "설계 명세와 실제 데이터의 괴리(drift)"를 없애는 것이 목표다.

## 2. 해결 기제와 기술 (Mechanism & tech)

**(1) 트래킹 플랜 = 협업형 스키마 명세**
웹 앱에서 이벤트(Events), 속성(Properties), 지표(Metrics), 소스(Sources), 목적지(Destinations)를 구조화된 형태로 정의한다. 전역 이벤트·속성 네임스페이스로 명명 일관성을 강제하고, 지속적인 "Tracking Plan audit(명명 규칙 감사)"를 돈다. 공식 문서는 JSON Schema를 쓴다고 명시하지 않으나, 구조화된 스키마 정의를 코드 생성·검증에 사용한다([Tracking Plan](https://www.avo.app/docs/data-design/avo-tracking-plan)). 다운스트림 스키마 레지스트리(Segment Protocols 등)로 자동 발행(auto publishing) 동기화도 지원한다([pricing](https://www.avo.app/pricing)).

**(2) Codegen — 타입 안전 코드 생성**
플랜의 각 이벤트마다 "이벤트 함수(Avo Function)"를 생성한다. 이 함수는 분석 SDK(GA4, Segment, Amplitude, Mixpanel, RudderStack, Snowplow, PostHog, 자체 API 등)를 감싸는 타입 안전 래퍼(wrapper)다. 개발자가 함수를 호출하면 코드가 (1) 전달된 속성 + 시스템 속성 수집 → (2) 플랜 제약 대비 검증 → (3) 설정된 각 목적지로 라우팅 → (4) 적절한 트래킹 메서드(log/identify 등) 실행을 자동 수행한다. 타입 안전은 두 층위:
- **컴파일 타임**: 언어 타입 시스템이 enum 제약, 필수/선택 속성 위반을 코드 실행 전에 잡음.
- **런타임**: JavaScript 같은 동적 언어에는 생성 코드에 런타임 체크를 넣어 타입·제약·필수 필드 위반을 로그/strict 모드로 보고.
개발자는 Avo CLI로 인증하고 생성 코드를 프로젝트에 pull한다([Codegen tech deep dive](https://www.avo.app/docs/implementation/avo-codegen-tech-deep-dive)).

**(3) Inspector — 런타임 스키마 검증**
앱에 SDK(npm/gradle/cocoapods)로 심는 런타임 도구. 핵심은 **값이 아니라 스키마 형태(타입)만 추출**한다는 것 — 예를 들어 user ID 값 `1337`은 `"int"` 타입으로 변환되어 서버로 전송된다(실제 데이터 미수집 → 개인정보 정책에 추가 항목 불필요). `trackSchema()`/`trackSchemaFromEvent()`를 이벤트 발생 시 호출하면 스키마를 로컬 배치 후 서버로 보내 플랜과 대조하고, 불일치는 서버 측에서 판정한다([Inspector overview](https://www.avo.app/docs/reference/avo-inspector-sdks/overview)).

**(4) 브랜치 워크플로 + 승인**
Git처럼 브랜치를 만들어 변경을 제안하고, Draft → Ready for review → Changes requested → Approved → Merged → Closed 상태로 진행한다. 리뷰어(데이터 담당·개발자)를 지정하며, 병합 시 다운스트림 스키마가 자동 발행된다([Approval workflows](https://www.avo.app/docs/data-design/branches/approval-workflows)).

**(5) Slack/알림 + CI/CD**
리뷰 요청 시 이메일 + (설정 시) Slack @멘션 알림, 리뷰 대기 브랜치 일일 리마인더. CI 파이프라인에 **avo status check**를 추가해, Codegen으로 보내야 할 이벤트가 코드베이스에 없으면 에러를 던지도록 구성할 수 있다([Integrate with other tools](https://www.avo.app/docs/publishing/integrations), [Validate](https://www.avo.app/docs/workflow/validate)). 트래킹 플랜 변경은 webhook으로도 구독 가능([webhook](https://www.avo.app/docs/implementation/avo-tracking-plan-webhook)).

## 3. 사람이 여전히 해야 하는 일

- **트래킹 플랜 자체를 사람이 작성**한다. 무엇을 추적할지, 이벤트/속성 구조, 명명은 PM·데이터 담당이 손으로 설계한다([Tracking Plan](https://www.avo.app/docs/data-design/avo-tracking-plan)).
- **개발자가 계측 호출을 손으로 삽입**한다. Avo는 함수를 생성해 줄 뿐, 코드베이스의 "어디에서, 언제" 이벤트를 발생시킬지는 개발자가 비즈니스 로직을 이해해 직접 결정하고 함수를 호출한다. 즉 Avo는 *how*(타입 안전 래퍼·라우팅)만 자동화하고 *where/when*은 사람 몫이다([devs-101](https://www.avo.app/docs/implementation/devs-101)).
- **QA/개발자가 화면을 직접 눌러 검증**한다. Inspector Debugger 검증은 사람이 "앱에서 이벤트를 트리거하고 실시간 스트림에서 확인"하는 수작업 테스트를 요구한다([Validate](https://www.avo.app/docs/workflow/validate)).
- **리뷰/승인**: 브랜치 리뷰어 지정·승인은 사람이 한다([Approval workflows](https://www.avo.app/docs/data-design/branches/approval-workflows)).

## 4. 미해결 영역·한계

**공식 문서상 전제 조건:**
- Codegen/Inspector를 쓰려면 각 소스에 **SDK를 설치하고 API 키를 설정**해야 한다([Inspector overview](https://www.avo.app/docs/reference/avo-inspector-sdks/overview)).
- 프로덕션 위반 감지는 **지난 24시간 데이터**를 보는 사후(after-the-fact) 방식이다([Validate](https://www.avo.app/docs/workflow/validate)).
- 강제형 승인(enforceable approvals), 자동 필수 리뷰 등 강한 거버넌스 기능은 **Enterprise 플랜 한정**([pricing](https://www.avo.app/pricing)).

**실사용자 불만(G2 직접 페이지는 403 차단, 검색 요약 기반):**
- **가파른 학습 곡선**: 비기술 팀원에게 진입 장벽, 초기 설정에 상당한 시간 투자 필요([G2 리뷰 요약](https://www.g2.com/products/avo/reviews?qs=pros-and-cons), [SoftwareAdvice](https://www.softwareadvice.com/data-governance/avo-profile/)).
- **Journey Planner가 아직 미완성**("not fully fledged yet")이라는 평([G2 리뷰 요약](https://www.g2.com/products/avo/reviews)).
- **커스텀 이벤트 명명 구조의 의미 검사 한계**: 표준 형식 외 커스텀 명명 규칙에 대한 시맨틱 체크 요청([G2 리뷰 요약](https://www.g2.com/products/avo/reviews)).
- 긍정: 투명성, git 유사 리뷰 구조, 역할 기반 접근, 반응 빠른 지원 — 예: Patreon은 기능 릴리스당 1~4일 걸리던 작업이 30~60분으로 단축.

## 6. 핵심 용어

- **Tracking Plan(트래킹 플랜)**: 어떤 이벤트·속성을 어떻게 수집할지 정한 단일 진실 명세
- **Event / Property**: 추적 단위와 그에 딸린 데이터 필드
- **Source / Destination**: 데이터를 보내는 앱·플랫폼 / 데이터를 받는 분석 도구
- **Codegen**: 플랜에서 타입 안전 이벤트 함수를 자동 생성
- **Inspector**: 앱 런타임에서 실제 이벤트의 스키마(타입)를 분석해 플랜과 대조
- **Branch(브랜치)**: Git처럼 플랜 변경을 격리해 제안·리뷰·병합하는 단위
- **Journeys**: 스크린샷/디자인 이미지로 사용자 흐름을 그리고 각 순간(트리거)을 이벤트에 연결하는 설계 기능
- **avo status check**: CI에서 계측 누락을 검사하는 체크

## 지정 질문 답변

**Q1. "개발자가 계측 코드를 손으로 짠다"는 전제 유지 여부** — 그렇다, 유지한다. Codegen이 함수를 만들어 주지만 개발자가 정확한 위치에 직접 호출을 삽입해야 한다. 미해결로 남는 것: (a) 계측 지점 누락 오류(함수를 안 부르면 Codegen은 모름 — CI status check가 부분 보완), (b) 코드에 함수를 심는 반복 노동 자체, (c) UI 요소와 이벤트의 실제 연결이 올바른지에 대한 자동 보장 부재.

**Q2. 플랜↔실데이터 괴리 탐지 시점** — 네 시점: ① 컴파일 타임(타입 시스템) ② CI(avo status check — 미계측 검출) ③ 개발/스테이징 런타임(Inspector Debugger — 사람이 눌러야 함) ④ 프로덕션 사후(Inspector Issues — 지난 24시간 집계). 실측 검증은 실제 이벤트 발생이 전제이며 개발 검증은 사람이 화면을 눌러 트리거해야 한다.

**Q3. 요소↔이벤트 매핑·시각 증빙** — 부분적으로, 설계 단계에서만. Journeys가 스크린샷/디자인 이미지에 주석을 달아 화면·액션 지점을 이벤트에 연결(AI 스크린샷 분석으로 트리거 자동 제안, 트리거별 코드 스니펫 생성)([Journeys](https://www.avo.app/docs/data-design/avo-tracking-plan/journeys), [Figma→Tracking Plan](https://www.avo.app/blog/from-figma-to-tracking-plan-in-minutes)). 그러나 이는 설계 의도 매핑이며, 런타임에서 "그 버튼을 눌렀더니 이 이벤트가 실제로 나왔다"는 시각 증거를 자동 대조·증빙하지는 않는다. Inspector는 스키마 타입만 보므로 UI 요소↔실측 이벤트의 자동 결합은 Avo의 사각지대.
