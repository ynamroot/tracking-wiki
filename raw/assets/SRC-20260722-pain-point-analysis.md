# 행동데이터 생애주기 — 역할별 페인포인트 분석

> 목적: 트래킹 플랜 자동화 도구 PRD의 문제 정의를 뒷받침하는 실무자 고통 분석.
> 근거: 외부 1차 자료(실무 블로그·업계 리포트·도구사 조사, 출처 URL 표기) + 내부 실증(가치제안 문서의 [실증] 항목).
> 원자료: `raw/페인포인트-조사.md`

---

## 요약 — 가장 자주 확인된 고통 톱5

| 순위 | 고통 | 누구의 고통 | 근거 강도 |
|---|---|---|---|
| 1 | 태그 파손을 "대시보드가 이상해서" 뒤늦게 발견 — 며칠에서 수개월 방치 | 데이터매니저, 리더십 | 4개 이상 출처 독립 언급 + 내부 실증 일치 |
| 2 | 명명 규칙 난립 + 문서와 실제의 괴리 — 이름 변경 이력 부재 | 데이터매니저, 기획자 | 국내외 다수 출처 + 내부 실증 일치 |
| 3 | 계측 리드타임 — 기능은 나갔는데 데이터는 2주 뒤 | 기획자/PM | 복수 출처 |
| 4 | 수동 QA·스프레드시트 수작업의 반복 — 조직이 클수록 폭증 | 데이터매니저, FE | 국내 사례(원티드) 포함 복수 출처 |
| 5 | 데이터 신뢰 상실 → 조직이 데이터 자체를 회피 | 전 역할 | HBR·Forrester 수치 인용 |

이 다섯 가지는 서로 독립된 문제가 아니라 하나의 사슬이다: **수작업 의존(4) → 커버리지·품질 제한(2,3) → 파손 방치(1) → 신뢰 상실(5)**. 가치제안 문서의 "품질과 양이 담당자 리소스와 맞바꾸는 관계"라는 문제 정의는 이 사슬의 압축으로, 외부 자료와 정합한다.

## 1. 데이터매니저 (트래킹 플랜 운영 주체)

### 1-1. 파손의 침묵 — 이벤트가 조용히 멈춰도 아무도 모른다
- **고통**: 이벤트 하나가 멈춰도 전체 트래픽은 멀쩡해 보인다. 리포트를 열어본 시점에야 유실을 안다.
- **외부 근거**:
  - "며칠, 어쩌면 일주일 동안 이벤트 볼륨이 0인데 아무도 눈치채지 못한다. 사이트 트래픽 전체는 멀쩡해 보이기 때문" ([Trackingplan](https://www.trackingplan.com/blog/data-quality-monitoring)) — 같은 글이 "모니터링 없이는 매달 데이터의 15~25%가 손상될 수 있다"는 업계 조사를 인용.
  - "개발자가 앱 업데이트 중 login_event를 user_login으로 바꿨다. 로그인 대시보드는 0을 보여주지만 사용자는 로그인하고 있다" ([RudderStack 창업자](https://www.linkedin.com/posts/soumyadeb-mitra_most-data-teams-discover-broken-tracking-activity-7324071773392486401-bWSo))
  - "전환 이벤트가 실제 전환마다 두 번 발화되어 8개월간 리포트가 현실의 2배였다. 아무도 확인하지 않았다" ([House of Martech](https://houseofmartech.com/blog/systematic-event-tracking-why-your-data-keeps-breaking-and-how-to-fix-it))
- **내부 실증 대응**: 가치제안 문서 1장 "파손은 데이터가 이미 유실된 뒤에야 발견", "프로모션 성과 분석 요청을 받았으나 로그 QA가 끝날 무렵 프로모션이 종료" — 외부 자료와 정확히 같은 구조.
- **함의**: 파손 발견은 "능동 감시"가 아니라 "소비 시점의 우연"에 의존한다. 파손이 침묵하는 이유는 감시 주체가 없어서이지, 감지가 기술적으로 어려워서가 아니다.

### 1-2. 이벤트 사전이 화면을 설명하지 못한다
- **고통**: 로그 ID·이벤트 이름만으로는 화면의 어떤 요소인지 알 수 없다. 이름 변경 이력이 없어 과거 데이터와 연결이 끊긴다.
- **외부 근거**: "담당 PD마다 다르게 이벤트 로그를 작성", "snake_case와 camelCase가 뒤섞여 차이를 모르겠다", "이름이 바뀐 경우 이력이 적힌 곳이 없어 예전 데이터가 있음에도 트래킹하지 못한다" ([브런치 — 데이터 이벤트 로그 설계 과정](https://brunch.co.kr/@jinny/18))
- **내부 실증 대응**: 대응표 #2 "로그 ID만으로 화면의 어떤 요소인지 알 수 없어 사전·라벨링 유지가 별도 업무", #7 "명명 규칙 관리와 이름 짓기 자체가 업무".
- **함의**: 사전(데이터 딕셔너리)은 만들 때가 아니라 **유지할 때** 무너진다. 유지가 수작업인 한, 사전 최신성은 담당자 성실성의 함수가 된다.

### 1-3. 불려감의 반복 — 원인 추적과 책임 소재 불명
- **고통**: "대시보드가 깨진 것 같아요"라는 호출이 반복된다. 고칠 수 있는 사람에게 닿기까지 수동 단계가 여럿이고, 반복될수록 데이터 신뢰와 팀 사기가 깎인다.
- **외부 근거**: "고칠 수 있는 사람이 알림을 받기까지 세 개의 분리된 수동 단계", "반복될수록 대시보드에 대한 신뢰가 점차 상실", 상류 파손 뒤치다꺼리는 "상당히 사기를 꺾는다" ([incident.io](https://incident.io/blog/incident-management-for-data-teams))
- **내부 실증 대응**: 대응표 #5 "파손 원인 파악에 개발자와 왕복 커뮤니케이션".

### 1-4. 수동 QA의 반복 부담 — 조직이 커질수록 선형 초과 증가
- **고통**: 배포마다 이벤트가 제대로 심겼는지 사람이 확인한다. 스쿼드가 늘수록 이 시간이 같이 는다.
- **외부 근거**: "스쿼드와 프로젝트 수가 늘어남에 따라 이 작업에 투자하는 시간 또한 늘어났다" — 부담이 커서 QA 자동화 프로젝트를 별도로 착수 ([원티드랩 기술블로그](https://blog.wantedlab.com/library/insight/qa-automation))
- **내부 실증 대응**: 1장 "담당자가 감당할 수 있는 만큼만 계측되고, 그 이상은 포기된다".
- **함의**: 국내 대형 서비스도 같은 병목을 겪고, 그 해법으로 "검증 자동화"에 이미 투자하고 있다 — 문제의 실재성과 지불 의사의 방증.

### 1-5. 신뢰 상실의 비용 — 수치로
- **외부 근거** ([Mixpanel 블로그](https://mixpanel.com/blog/the-business-impacts-of-data-quality-issues/)):
  - "기업 데이터 중 품질이 좋은 것은 3%뿐" (HBR 인용)
  - "나쁜 데이터를 이해하는 데 분석가 시간의 40% 이상이 쓰인다" (Forrester 인용)
  - "KPI 불일치 논쟁이 의사결정보다 많은 시간을 소모"
  - 마케팅(오토캡처)과 개발(명시 계측)이 따로 계측해 서로 모순 → "자동 리포트는 아마 틀렸을 것"이라며 수동 스프레드시트로 예산 결정 ([House of Martech](https://houseofmartech.com/blog/systematic-event-tracking-why-your-data-keeps-breaking-and-how-to-fix-it))

## 2. 기획자 / PM (데이터 소비자이자 계측 요청자)

### 2-1. 배포 다음 날, 아무것도 안 찍히고 있다
- **외부 근거**: "출시 첫날: 경영진이 성과를 묻는다. 둘째 날: PM이 대시보드를 열고 깨닫는다 — 아무것도 추적되고 있지 않다", "결정적 출시 관측 시기를 놓치고, 최적화 결정을 깜깜이로 했다" ([The Analytics Tax](https://medium.com/@ori.o/the-analytics-tax-why-every-product-launch-requires-three-teams-and-two-weeks-cfacc5dadefc))
- **내부 실증 대응**: 1장의 프로모션 사례 — 분석 가능 시점에 프로모션이 이미 종료.
- **함의**: 기획자의 고통은 "데이터가 틀렸다"보다 **"필요한 시점에 없다"**가 먼저다. 리드타임이 곧 데이터의 가치를 결정한다.

### 2-2. 계측이 개발 우선순위에서 밀린다
- **외부 근거**: PM은 계측 검증에 막히고, 엔지니어는 기능 개발 중간에 계측을 끼워 넣느라 상시 문맥 전환, 분석팀은 계측 완료를 기다리는 병목 — "모든 출시가 3개 팀과 2주를 잡아먹는다" (The Analytics Tax, 위)

### 2-3. 이벤트 이름을 읽을 수 없다
- **외부 근거**: "Signup Completed, Signed Up, user_signup 같은 지저분한 라벨 때문에 팀이 분석 도구 사용 자체를 회피한다" (Mixpanel 블로그, 위) / 사내 사용자들의 "이 데이터 이름이 뭐예요?" 반복 질문 (브런치, 위)
- **함의**: 데이터 사전의 소비자는 데이터매니저가 아니라 기획자다. 사전이 화면·행동을 직관적으로 설명하지 못하면, 질문은 전부 담당자에게 돌아온다.

## 3. 프론트엔드 개발자 (계측 구현 주체)

### 3-1. 명세는 스프레드시트, 구현은 복사-붙여넣기
- **외부 근거**: "트래킹 플랜만 잘 짜면 로그 개발은 슉슉 되는 줄 알았다. 하지만.." — 하드코딩 문자열 오타, 구글시트를 보며 수동 복붙(자동완성 없음), 버전 관리 공백, 시트 코멘트와 Jira에 흩어진 진행 상황 ([티스토리 — 이벤트 로그 설계 +Ampli CLI](https://sowhatmylifeismine.tistory.com/261))
- **함의**: 개발자의 고통은 계측 자체가 아니라 **명세와 코드 사이의 수동 변환**이다. 이 글의 필자는 코드 생성(Ampli CLI)으로 해소했다 — "점(.)을 찍으면 이벤트·프로퍼티가 자동 완성된다".

### 3-2. 내 리팩토링이 계측을 깨뜨리는데, 나는 모른다
- **외부 근거**: "누군가 페이지를 다시 디자인하면 트래킹 코드가 참조하던 버튼이 사라져 이벤트가 멈춘다", "계측을 만든 개발자가 떠나면 새 개발자는 그 로직을 이해하지 못한다", "크게 터지기 전까지 팀들은 트래킹 얘기를 거의 하지 않는다" (House of Martech, 위) / 이벤트명 변경으로 파손을 만든 개발자 본인이 무인지 (RudderStack, 위)
- **함의**: 개발자에게 계측 파손은 **보이지 않는 부수효과**다. 파손 방지를 개발자의 주의력에 맡기는 설계는 구조적으로 실패한다. 책임을 옮길 게 아니라 감지를 자동화해야 한다.

### 3-3. 계측은 비즈니스 로직과 무관한 잡무
- **외부 근거**: 배포마다 수동 확인 반복, E2E 테스트로 잡으면 "이슈 발생 시 어디서 깨졌는지 디버깅이 어려움" (원티드랩, 위)

## 5. PRD를 위한 시사점

1. **문제의 일반성 확인**: 가치제안 문서의 [실증] 항목(표본 2)은 전부 외부 자료에서 독립적으로 재확인된다. "우리 조직만의 문제"가 아니라 업계 구조 문제다. 문서 7장의 표본 한계 우려는 이 조사로 상당 부분 보강 가능.
2. **고통의 사슬 구조**: 수작업 의존 → 커버리지 제한 → 파손 방치 → 신뢰 상실. 어느 한 지점만 풀면(예: 명명만 자동화) 사슬이 유지된다. 가치제안이 "하나의 순회가 부담 제거와 품질 보증을 동시 산출"을 내세우는 것은 이 사슬을 한 기제로 끊겠다는 주장으로, 사슬 구조와 맞물린다.
3. **역할별 가치 서술의 근거**: 기획자에게는 리드타임(2-1), 개발자에게는 무인지 파손의 자동 감지(3-2), 데이터매니저에게는 감시 주체의 존재(1-1)가 각자의 1순위 고통에 대응한다.
4. **국내 검증 사례**: 원티드랩이 같은 병목(배포마다 수동 이벤트 QA)에 자동화 투자로 답했다 — 문제의 지불 의사를 보여주는 국내 선례.
