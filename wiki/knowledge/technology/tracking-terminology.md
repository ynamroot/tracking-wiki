# 트래킹 상세·벤더 용어 사전

## Current Synthesis

이 문서는 [[key-terms|핵심 용어 해설]]의 기본 용어(이벤트, 속성, 스키마, 검증, 오토캡처 등)를 안다는 전제에서, 스키마·검증, 옵저버빌리티, 요소 식별, 데이터 계약·거버넌스, 위반 처리·파이프라인 운영 계층에 나오는 **상세 용어와 벤더 고유 용어**를 도메인별로 정리한다. 기본 용어가 낯설면 먼저 [[key-terms|핵심 용어 해설]]을 본다. 이 페이지는 제품 결론을 새로 제안하지 않고, 전문 용어를 통일해 다른 지식 페이지를 읽기 위한 참조 역할을 한다. <sup>[🔗](#source-1)</sup>

## Context

이 Wiki는 코드 생성, 스키마 검증, 데이터 옵저버빌리티, 요소 식별, 데이터 계약, 위반 처리를 함께 다뤄 벤더마다 이름이 다른 개념이 자주 겹친다. 예를 들어 코드 생성은 도구마다 Codegen, Ampli, Typewriter, Snowtype, Smartype, RudderTyper로 불리고, "검증"도 코드 작성 전·수집 시점·창고 적재 후로 위치가 갈린다. 이 문서는 그 이름과 위치를 한곳에서 대조하게 한다.

## Evidence

- 아래 정의는 내부 조사 8종에서 도출된 용어집을 근거로 하며, 그 원자료는 이미 이 Wiki에 accepted Source로 반영돼 있다. 외부 웹 리서치로 새로 만든 정의가 아니다. <sup>[🔗](#source-1)</sup>
- 벤더 고유 기능 이름과 표준(SchemaVer, Iglu, ODCS, dbt model contracts 등)은 용어집이 각 도구·표준에 귀속시켜 구분한다. <sup>[🔗](#source-1)</sup>

## Mechanics

### 스키마·검증

| 용어 | 쉬운 설명 |
|---|---|
| JSON Schema | 데이터 구조를 기계가 읽을 수 있게 적은 규격. 트래킹 검증의 사실상 표준이다. |
| 스키마 레지스트리(schema registry) | 스키마를 한곳에 저장하고 파이프라인 곳곳에 나눠 주는 창구. Snowplow에선 Iglu가 이 역할을 한다. |
| 자기기술 이벤트(self-describing event) | 데이터 안에 "나는 이 스키마를 따른다"는 주소를 함께 담아, 데이터가 자기 규격을 들고 다니게 만든 이벤트(Snowplow의 `iglu:` 주소). |
| SchemaVer | 스키마 전용 버전 표기(`1-0-0`). 깨는 변경=major, 부분 호환=minor, 완전 호환=patch로 구분한다(Snowplow). |
| 스키마 진화(schema evolution) | 과거 데이터를 깨뜨리지 않으면서 스키마를 바꿔 가는 것(예: 선택 필드 추가). |
| 컴파일 타임 / 런타임 검증 | 컴파일 타임은 코드 실행 전 타입 시스템이 오류를 잡는 것, 런타임은 실행 중 값·타입·필수 필드 위반을 잡는 것. 코드 생성 도구는 대개 두 층을 함께 쓴다. |
| AJV | 자바스크립트에서 JSON Schema를 런타임에 검사하는 라이브러리. Segment Typewriter의 JS/TS 클라이언트가 잘못된 호출을 크래시 대신 경고로 남기는 데 쓴다. |
| 데이터 플랜·데이터 포인트(data plan / data points) | mParticle의 트래킹 플랜에 해당. 플랜 하나에 데이터 포인트(검증 단위) 최대 1,000개를 두고 각 포인트를 JSON Schema로 검증한다. |
| Data Structures 승격 | Snowplow에서 스키마를 초안(draft)→개발(development)→운영(production)으로 단계 승격시키는 절차. 초안은 검증에 쓰이지 않는다. |
| CI 계측 누락 검사 | 배포 파이프라인(CI)에 붙여, 플랜에는 있는데 코드에 안 심긴 이벤트가 있으면 막는 검사(Avo `avo status`, Amplitude `ampli status`). |

### 감지·모니터링·옵저버빌리티

| 용어 | 쉬운 설명 |
|---|---|
| 데이터 옵저버빌리티(data observability) | 스키마를 미리 강제하지 않고, 이미 쌓인 데이터가 평소와 다르게 행동하는지를 지속 감시해 사고를 조기에 알리는 접근. |
| 다섯 기둥(five pillars) | 데이터 건강을 재는 다섯 축: 신선도·볼륨·품질(분포)·스키마·계보(Monte Carlo). |
| 계보(lineage) | 데이터가 어디서 와서 어디로 흐르는지의 연결. 단, 대개 웨어하우스 안까지만 이어지고 앱 코드·화면까지는 못 내려간다. |
| 기준선 학습(ML baseline) | 머신러닝이 데이터의 "정상 패턴"을 자동으로 익혀 정상 범위를 세우고, 벗어나면 알리는 방식. |
| 이상 감지·그레이 밴드(anomaly detection / grey band) | 학습한 정상 패턴과 계속 대조해 벗어남을 잡되, 계절·캠페인에 따라 정상 범위를 동적으로 넓혀 오경보를 줄이는 것(Trackingplan). |
| 알림 피로·오탐(alert fatigue / false positive) | 쓸모없는 알림이 많아 담당자가 무뎌지는 것, 문제가 없는데 울리는 알림. 옵저버빌리티 도입 초기의 대표 과제. |
| 스키마 드리프트 / 트래킹 플랜 드리프트 | 실제 이벤트 구조가 명세에서 시간이 지나며 벗어나는 것 / 문서상 계획과 실제 구현·발생 이벤트가 어긋나는 것. |
| 딥 오딧(deep audit) | 상시 감시와 별도로, 매일 한 번 세션 단위(깨진 퍼널 순서·중복·도구 간 불일치)를 배치로 점검하는 것(Trackingplan). |
| 구현 상태(implementation status) | 플랜 대비 실제 발생 이벤트가 얼마나 구현·정합됐는지 신호등처럼 표시하는 것(Avo Inspector). |

### 오토캡처·요소 식별

| 용어 | 쉬운 설명 |
|---|---|
| 소급 분석(retroactive analytics) | 원시 데이터를 일단 다 모아 두었다가 나중에 정의를 만들어 과거까지 거슬러 분석하는 방식. 오토캡처 계열의 강점. |
| 버추얼/라벨드 이벤트(virtual / labeled event) | 원시 오토캡처 데이터를 가리키는 사람이 읽을 이름. 나중에 붙여도 과거 데이터에 소급 적용된다(Heap Virtual Event = Amplitude Labeled Event). |
| 비주얼 태깅/라벨링(visual tagging / labeling) | 화면 위에서 요소를 클릭해 이벤트를 정의하는 것. 속으로는 CSS 셀렉터를 자동 생성한다. |
| 액션(action) | PostHog에서 셀렉터·텍스트·URL로 사후 매칭해 만드는 이벤트 묶음. 소급 적용된다. |
| 콤보 이벤트·이벤트 리페어(combo event / event repair) | 화면 개편으로 정의가 매칭을 멈췄을 때, 구·신 버전을 합치거나 정의만 갈아 끼워 코딩 없이 추적을 복원하는 Heap의 장치. |
| DOM 스크레이핑(DOM scraping) | 데이터레이어 없이 CSS 셀렉터로 화면에서 값을 직접 긁어 오는 우회법. 화면 개편 시 파손된다. |
| data-* 속성 | 개발자가 요소에 미리 붙여 두는 안정적 식별용 표식. 셀렉터가 잘 안 깨지도록 앵커로 삼으라고 여러 도구가 권한다. |
| 요소 지문(element fingerprint) | 한 요소를 여러 속성 묶음(셀렉터·class·name·aria-label·DOM 위치·보이는 텍스트·주변 요소)으로 식별해, 화면이 바뀌어도 같은 요소로 인식하는 방식. |
| 자가치유 셀렉터(self-healing selector) | 셀렉터가 깨졌을 때 대체 위치 지정자를 자동으로 찾아 잇는 기술. 테스트 자동화 진영(Testim·Healenium 등)에 있고, 태그 감사 도구에는 대체로 없다. |
| 속성 안정성 순위 | 요소 식별 신호의 신뢰 순서. 대체로 접근성 속성(ARIA) > 보이는 텍스트 > name/placeholder > CSS 클래스 > DOM 위치 > 자동 생성 ID. |
| 신뢰도 점수(confidence scoring) | 바뀐 화면에서 찾은 후보가 원래 지문과 얼마나 일치하는지 매긴 점수. 높으면 자동 적용, 낮으면 사람이 검토. |
| 결정론적 실행(deterministic execution) | "AI는 테스트 생성·유지보수만 하고 실행은 정해진 코드로 한다"는 원칙. 재현성·비용 통제를 위한 것(Octomind). |
| 데드 클릭·레이지 클릭 | 눌러도 반응 없는 클릭 / 답답해 연타하는 클릭. 오토캡처가 추가로 잡아 주는 UX 신호. |
| 미계측 요소(untracked element) | 상호작용은 되는데 이벤트 태그가 없는 요소. 이를 능동적으로 자동 발견하는 도구는 시장에 사실상 없다는 것이 원자료의 결론. |

### 태그매니저

| 용어 | 쉬운 설명 |
|---|---|
| 태그·트리거·변수·컨테이너 | 태그=실행 단위(예: GA4 전송), 트리거=발화 조건, 변수=값 추출기, 컨테이너=이들을 담는 설정 묶음. |
| 요소 가시성 트리거 | 요소가 뷰포트에 N%·M초 이상 보이면 발화하는 트리거. 노출 추적에 쓴다. |
| 히스토리 변경 트리거 | 페이지를 새로 안 불러오는 SPA에서 주소(라우팅) 변경을 감지해 발화하는 트리거. |
| 커스텀 JS 변수 / DOM 요소 변수 | 데이터레이어에 없는 값을 코드나 셀렉터로 화면에서 긁어 오는 변수. 편하지만 마크업이 바뀌면 멈춘다. |
| 프리뷰 모드·태그 어시스턴트 | 발행 전에 사람이 직접 페이지를 돌며 태그 발화·데이터레이어 상태를 확인하는 수동 디버깅 세션. 상시 자동 감시는 아니다. |
| 컨테이너 부채·태그 비대 | 죽은 태그·중복 스크립트가 쌓여 방치된 상태. 6개월이면 다시 비대해지는 건 거버넌스 문제라는 지적이 있다. |
| 동의 정책·PII·피기배킹 | 지역별 수집 허용 규칙 / 개인식별정보 / 서드파티 태그가 몰래 다른 태그를 붙이는 것. 태그 감사 도구의 프라이버시 감시 대상. |

### 데이터 계약·거버넌스

| 용어 | 쉬운 설명 |
|---|---|
| 데이터 계약(data contract) | 데이터를 만드는 쪽과 쓰는 쪽이 스키마·의미·품질·SLA·소유권을 못박은, 버전 관리되는 공식 합의. |
| 생산자·소비자(producer / consumer) | 데이터를 만들어 내보내는 쪽과 받아 쓰는 쪽. 둘 사이 명시적 합의 부재가 많은 데이터 사고의 근본 원인으로 지목된다. |
| shift-left | 품질 책임을 하류(소비자가 나중에 청소)에서 상류(생산자가 원천에서 보증), 즉 코드 리뷰·CI 등 더 앞 단계로 옮기는 것. |
| ODCS(Open Data Contract Standard) | 데이터 계약을 적는 업계 표준 서식. Bitol이 관리하는 Linux Foundation 프로젝트. |
| dbt model contracts | dbt가 데이터 모델을 빌드하는 시점에 계약을 검증하는 기능. 데이터가 이미 변환된 뒤라 명백히 사후 검증이다. |
| 거버넌스 사전형 장치 | 유입을 막는 강제 검증이 아니라 사후에 신뢰도를 표시·검토하는 장치. Verified Data(신뢰 배지), Data Standards(이름 규칙 준수 표시), Event Approval(신규 이벤트 검토)이 대표(Mixpanel). |
| 데이터 사전(data dictionary) | 수집 중인 이벤트·속성 목록과 뜻풀이 문서. Mixpanel Lexicon이 대표로, 강제 게이트라기보다 서술적 카탈로그다. |
| 선언적 태깅(declarative tagging) | HTML 마크업에 속성(예: `data-elb=...`)으로 수집 규칙을 직접 새겨 매핑이 코드에 드러나게 하는 방식(walkerOS). |
| 하이브리드 계측(hybrid tracking) | 오토캡처로 기본선을 깔고 핵심 지표 10~20개는 구조화 속성과 함께 수동 계측하는 절충. 2024년 이후 업계가 수렴한 방식. |
| 서버사이드 계측 | 브라우저·앱이 아니라 서버에서 이벤트를 보내는 방식. 핵심 지표는 이렇게 심으라는 실무 합의가 있다. |

### 위반 처리·파이프라인 운영

| 용어 | 쉬운 설명 |
|---|---|
| 위반(violation) | 이벤트·속성이 트래킹 플랜과 어긋난 사건. 흔한 유형은 계획에 없는 이벤트·속성, 타입 불일치, 필수 필드 누락. |
| 차단(blocking) / 차단 이벤트 | 위반이라서 파이프라인에서 드롭하는 처리, 또는 그렇게 버려진 이벤트. 격리해 두지 않으면 영구 복구가 안 된다는 경고가 반복된다. |
| 전파(propagate) | 위반 이벤트를 버리지 않고 에러 표시를 붙인 채 목적지까지 흘려보내는 처리. RudderStack 기본값이라 유실 위험이 낮은 대신 나쁜 데이터가 하류까지 간다. |
| 격리(quarantine) | 차단·실패한 데이터를 버리지 않고 별도로(예: S3) 보관해 나중에 복구할 수 있게 하는 것(mParticle Quarantine Connection). |
| 실패 이벤트(failed events) | 스키마 검증·강화 등에서 실패해 깨끗한 테이블에 못 들어가고 별도 격리·백업되는 이벤트(Snowplow). 재처리할 수 있다. |
| 데드레터 큐(DLQ) | 처리에 실패한 메시지를 버리지 않고 따로 모아 두는 대기열(Kafka 등). |
| 보강·강화(enrichment) | Snowplow에서 수집 이벤트를 웨어하우스에 넣기 전 스키마 검증·부가정보 추가를 하는 단계. 검증이 여기서 이뤄진다. |
| 변환(transformation) | 코드를 고치지 않고 이벤트·속성의 이름·값을 바로잡는 처리(Segment). |
| 재처리·백필(replay / backfill) | 격리한 실패 이벤트를 다시 태워 복구하는 것 / 빠졌던 과거 데이터를 나중에 채워 넣는 것. 둘 다 대체로 수동이라 손이 많이 간다. |
| MTU·MAU | 도구 과금의 흔한 기준 단위(월간 추적/활성 이용자). 위반을 별도 소스로 전달하면 MTU가 더 소모되기도 한다. |

각 표는 비교가 아니라 참조용이라 "그래서 무엇을 뜻하는가"를 개별 항목의 풀이에 담았다. 요약하면, 이 용어들은 대부분 **검증·복구가 수집 지점보다 뒤(창고·파이프라인 하류)에서 일어나거나, 사람이 규칙·범위를 손으로 정의해야 한다**는 공통 성질을 보인다. 이 성질이 [[validation-layer-model|검증 계층 모델]]과 [[automation-opportunity|자동화 기회]]에서 말하는 상류 증거 공백과 연결된다.

## Evaluation Criteria

- 새 지식 페이지가 벤더 고유 용어를 쓰면 이 사전의 귀속(어느 도구·표준의 개념인지)을 따른다.
- 기능·가격·지원 여부 같은 확정 claim은 이 사전이 아니라 각 벤더의 primary Source로 검증한다. 이 사전은 뜻풀이 참조까지만 담당한다.

## Contradictions

용어집은 외부 웹 리서치 없이 원자료의 정의만 정리했으므로 벤더별 미묘한 차이나 최신성까지 보장하지 않는다. 특히 "검증"은 코드 작성 전·수집 시점·창고 적재 후를 모두 가리킬 수 있어, 이 사전을 인용하는 문서는 검증 위치를 함께 적어야 한다.

## Open Questions

- 문서가 늘며 새 벤더 용어가 생기면 이 사전과 [[key-terms|핵심 용어 해설]]에 함께 반영해야 한다. 이 항목은 [[open-questions|미결 질문]]의 정식 `OQ`는 아니며, 운영상 유지보수 메모다.

## Product Implications

요소 지문·자가치유·신뢰도 점수·결정론적 실행은 제품 차별화 축이 될 개념이다. 이들을 이 사전에서 통일된 뜻으로 고정해 두면, 기능 정의와 기술 스펙에서 같은 개념을 다른 말로 부르는 혼선을 줄일 수 있다.
