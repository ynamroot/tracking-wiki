# 자료 요약: 행동데이터 운영 실무자 페인포인트

## 현재 종합

이 자료는 행동데이터 운영이 단순히 SDK(개발자가 앱에 붙이는 수집 도구)를 심는 문제가 아니라, 기획자(PM), 개발자, 데이터·분석(analytics) 담당자, 품질검사(QA)가 반복해서 맞물리는 운영 문제임을 보여 준다. 핵심 병목은 트래킹 플랜(tracking plan) 작성, 구현 티켓 만들기, 코드 삽입, 릴리즈 전 품질검사, 릴리즈 후 파손 감지, 하류(downstream) 신뢰 회복이 서로 떨어져 있다는 데 있다. <sup>[🔗](#source-1)</sup>

## 근거

- 행동데이터 파손은 “이벤트가 안 온다”보다 넓은 개념이다. 이벤트 이름이 바뀌는 어긋남(drift), 속성 타입 오류, 필수 속성 누락, 중복 이벤트, 목적지별 변환 오류, 태그·동의 조건별 발생 차이가 모두 운영 비용을 만든다. <sup>[🔗](#source-1)</sup>
- 기획자와 데이터 담당자는 무엇을 측정해야 하는지 알고 있어도, 코드 삽입과 릴리즈 검증은 개발자·품질검사 대기열(queue)에 의존한다. <sup>[🔗](#source-1)</sup>
- 내부 조사 자료의 정량 주장(claim)에는 벤더 블로그와 2차 인용이 섞여 있으므로, 구매자·예산·시간 비용은 고객 인터뷰나 사고(incident) 기록으로 다시 검증해야 한다. <sup>[🔗](#source-1)</sup>

## 작동 방식

작업 흐름은 대체로 `분석 질문 -> 트래킹 플랜 -> 개발 티켓 -> 코드 삽입 -> 품질검사 -> 배포 -> 대시보드 이상 발견 -> 원인 추적` 순서다. 이 흐름에서 앞단의 의도와 뒷단의 실제 발생 데이터가 직접 연결되지 않으면, 각 팀은 서로 다른 증거를 본다. 기획자는 요구사항 문서를 보고, 개발자는 코드와 SDK 타입을 보고, 데이터 담당자는 데이터 웨어하우스(warehouse)나 분석 도구의 결과를 본다.

## 평가 기준

- 고객 인터뷰에서는 “누가 고통을 느끼는가”와 “누가 돈을 내는가”를 분리해야 한다.
- 현재 비용은 업무 시간, 릴리즈 지연(release delay), 사고 건수, 대시보드 신뢰 하락, 하류 분석 재작업으로 측정한다.
- 고통의 심각도(pain severity)는 빈도보다 “발견 지연”과 “수정까지 필요한 부서 간 인계(cross-functional handoff) 수”를 함께 봐야 한다.

## 모순

내부 자료는 여러 공개 자료를 종합했지만 실제 고객 원문은 아니다. 경제적 구매자와 초기 챔피언(champion)을 확정하기에는 근거 수준이 부족하다.

## 미결 질문

- `OQ-001`: 경제적 구매자와 초기 챔피언은 누구인가?
- `OQ-004`: 수동 품질검사·파손 대응·계측 구현 비용은 얼마인가?

## 제품 시사점

초기 제품은 “계측을 더 쉽게 작성”보다 “의도, 구현, 실제 발생 사이의 증거 사슬(chain)을 줄이는 것”에 초점을 맞춰야 한다. 구매자는 데이터 리더일 수 있지만, 챔피언은 릴리즈 품질검사 담당이나 기획자일 가능성을 별도로 검증해야 한다.
