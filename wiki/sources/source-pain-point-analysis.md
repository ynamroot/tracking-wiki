# Source Summary: 역할별 페인포인트 분석

## Current Synthesis

이 Source는 행동데이터 계측의 고통을 데이터매니저·기획자/PM·프론트엔드 개발자 세 역할로 나누고, 각 고통을 외부 1차 자료(실무 블로그·업계 리포트)와 내부 실증(가치제안 문서)으로 교차 검증한 분석 보고서다. 핵심 기여는 두 가지다. 첫째, 같은 프로젝트의 기존 내부 조사([[source-practitioner-pain-points|행동데이터 운영 실무자 페인포인트]])가 "모두 secondary"라 지적됐던 부분에, 이름이 있는 복수의 외부 자료를 붙여 문제의 일반성을 보강한다. 둘째, 흩어진 고통을 하나의 사슬(수작업 의존 → 커버리지·품질 제한 → 파손 방치 → 신뢰 상실)로 묶어 설명한다. <sup>[🔗](#source-1)</sup>

## Evidence

- 파손의 침묵: 이벤트 하나가 멈춰도 전체 트래픽은 멀쩡해 보여 소비 시점에야 발견된다. Trackingplan, RudderStack 창업자, House of Martech가 각각 독립적으로 같은 구조(로그인 이벤트명 변경으로 대시보드 0, 전환 이벤트 이중 발화로 8개월간 2배)를 보고한다. <sup>[🔗](#source-1)</sup>
- 수동 QA 부담: 원티드랩 기술블로그는 스쿼드·프로젝트 수가 늘수록 이벤트 QA 시간이 함께 늘어 별도 자동화 프로젝트를 착수했다고 밝힌다. 국내 대형 서비스도 같은 병목을 겪고 자동화에 투자한 사례로, 문제의 실재성과 지불 의사를 함께 보여준다. <sup>[🔗](#source-1)</sup>
- 개발자 무인지 파손: 리팩토링·화면 개편으로 참조하던 버튼이 사라져 이벤트가 멈춰도 만든 개발자 본인이 모른다. 파손 방지를 개발자 주의력에 맡기는 설계는 구조적으로 실패한다는 것이 저자의 함의다. <sup>[🔗](#source-1)</sup>
- 신뢰 상실 비용(secondary 인용): Mixpanel 블로그가 HBR을 인용해 "품질 좋은 데이터는 3%뿐", Forrester를 인용해 "나쁜 데이터 이해에 분석가 시간의 40% 이상"을 든다. 이는 벤더 블로그가 제3자를 재인용한 secondary 수치다. <sup>[🔗](#source-1)</sup>

## Mechanics

분석은 역할별로 고통을 나열한 뒤, 각 항목에 외부 근거와 내부 실증(가치제안 문서의 대응표 항목·프로모션 사례)을 나란히 붙이는 방식으로 전개한다. 마지막에 역할 간 협업 마찰(언어 차이, 네이밍 논쟁, 책임 소재 불명, 핸드오프 세금, 도구 이원화)을 표로 정리하고, PRD를 위한 시사점 네 가지로 맺는다.

## Evaluation Criteria

- 역할별 1순위 고통이 서로 다르다: 기획자=리드타임("필요한 시점에 없다"), 개발자=무인지 파손의 자동 감지, 데이터매니저=감시 주체의 부재. 제품 가치 서술은 이 대응에 맞춘다.
- 고통의 사슬 구조상 한 지점만 풀면(예: 명명만 자동화) 사슬이 유지된다. 단일 기제로 사슬을 끊는다는 주장이 검증 기준이 된다.

## Contradictions

이 자료는 secondary 분석이다. 인용된 외부 URL은 문제의 일반성을 보강하지만, 정량 수치(3%, 40%, 15~25%)는 벤더 블로그가 HBR·Forrester·업계 조사를 재인용한 것이라 원 조사까지 provenance를 확장해야 확정할 수 있다. 또한 인용 사례는 대체로 서구 도구사·블로그와 국내 블로그이며, 우리 타깃 고객의 실제 비용을 직접 측정한 것은 아니다.

## Open Questions

- `OQ-001`: 경제적 구매자와 초기 champion은 누구인가? 이 자료는 역할별 고통은 보이지만 지불 주체는 확정하지 못한다.
- `OQ-003`: Value Proposition 검증에 필요한 고객 인터뷰·행동 증거 기준은 무엇인가?
- `OQ-004`: 수동 QA·파손 대응·계측 구현의 실제 시간·비용은 얼마인가? 외부 자료는 문제 존재를 보강하나 타깃 고객의 비용 수치는 주지 않는다.

## Product Implications

문서 7장의 "우리 조직만의 문제일 수 있다"는 표본 한계 우려는 이 분석으로 상당 부분 보강된다. 초기 메시지는 역할별 1순위 고통에 맞춰 분기하되(기획자=리드타임, 개발자=무인지 파손 감지, 데이터매니저=감시 주체), 실제 판매 motion과 비용 근거는 buyer/champion·비용 open question이 풀린 뒤 확정한다.
